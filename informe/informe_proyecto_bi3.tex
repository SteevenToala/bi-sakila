\chapter*{Resumen}
\addcontentsline{toc}{chapter}{Resumen}
\thispagestyle{fancy}

El presente informe documenta el diseño, implementación y análisis del \textbf{Proyecto BI3}, cuyo objetivo es construir un modelo de inteligencia artificial capaz de apoyar la toma de decisiones comerciales en una videotienda. La fuente de datos es una base de datos \textbf{MongoDB} (\texttt{BI\_Final}) que registra transacciones históricas de alquiler de películas.

El pipeline completo comprende cinco etapas: (1) extracción de datos desde MongoDB, (2) ingeniería de características y etiquetas con escalado estadístico, (3) análisis exploratorio de datos (EDA) con visualizaciones estadísticas, (4) entrenamiento comparativo de cinco arquitecturas de redes neuronales de aprendizaje multitarea (\ingles{Multi-Task Learning}), y (5) despliegue de una aplicación web interactiva con Flask.

El modelo final predice simultáneamente \textbf{ocho variables objetivo}: cuatro etiquetas binarias (ingreso alto, renta en fin de semana, preferencia de género, renta prolongada) y cuatro etiquetas multiclase de tres categorías (rango de precio, grupo de edad, nivel de fidelidad, popularidad de película). La entrada al modelo comprende 31 características numéricas resultado de escalar y codificar diez variables continuas, dieciséis variables de género cinematográfico (OHE) y cinco de clasificación de edad (OHE).

\vspace{0.5em}
\noindent\textbf{Palabras clave:} redes neuronales, aprendizaje multitarea, análisis exploratorio, MongoDB, Flask, clasificación binaria, clasificación multiclase.

\section{Contexto y motivación}

Las videotiendas generan volúmenes significativos de datos de transacción que, en su forma cruda, aportan escaso valor estratégico. La inteligencia de negocios permite transformar esos registros en información accionable: anticipar el comportamiento de alquiler, segmentar a los clientes, optimizar el catálogo y diseñar campañas de fidelización dirigidas.

El presente proyecto aplica técnicas de \ingles{machine learning} supervisado —concretamente redes neuronales profundas con múltiples salidas— para predecir de forma simultánea varias características del comportamiento de alquiler a partir de un único vector de entrada. Este enfoque, denominado \ingles{Multi-Task Learning} (MTL), comparte representaciones internas entre tareas relacionadas, lo que puede mejorar la generalización respecto a modelos independientes por tarea.

\section{Objetivos}

\begin{itemize}
  \item \textbf{Objetivo general:} Construir un sistema de inteligencia de negocios basado en una red neuronal multitarea que prediga simultáneamente ocho indicadores comerciales de alquiler de películas.
  \item \textbf{Objetivos específicos:}
  \begin{enumerate}
    \item Extraer y limpiar datos desde MongoDB hacia un dataset tabular analizable.
    \item Realizar un análisis exploratorio exhaustivo para caracterizar la distribución y las relaciones entre variables.
    \item Diseñar y comparar cinco arquitecturas de redes neuronales, seleccionando la de mayor precisión promedio.
    \item Desplegar una interfaz web que permita obtener predicciones en tiempo real.
  \end{enumerate}
\end{itemize}

\section{Alcance}

El sistema opera sobre datos históricos de la base \texttt{BI\_Final} almacenada en MongoDB. No incluye reentrenamiento en línea ni actualización automática del modelo. La aplicación web cubre el flujo de predicción individual para una transacción de alquiler dada.

\subsection{Cálculo de variables derivadas}

\begin{description}
  \item[\variable{veces\_alquilada\_pelicula}:] Conteo de frecuencia del identificador de película en todo el dataset. Mide la popularidad absoluta del título.

  \item[\variable{alquileres\_por\_cliente}:] Número total de transacciones registradas para cada \texttt{cliente\_ref}. Sirve como proxy del nivel de fidelidad.

  \item[\variable{variedad\_generos\_cliente}:] Número de géneros cinematográficos distintos que el cliente ha consumido, calculado con \texttt{groupby(\allowbreak"cliente\_ref")["categoria\_nombre"].transform("nunique")}.

  \item[\variable{pelicula\_popularidad\_actores}:] Primero se contabiliza cuántas veces aparece cada actor en el dataset completo (frecuencia absoluta). Luego, para cada película, se promedia la popularidad de todos los actores de su elenco.
\end{description}

\section{Escalado y codificación}

El pipeline de preparación aplica dos estrategias de escalado y una de codificación:

\begin{enumerate}
  \item \textbf{Estandarización Z-score} (media 0, desviación estándar 1) para las cinco variables con distribuciones libres o con valores extremos: \variable{ingreso}, \variable{veces\_alquilada\_pelicula}, \variable{alquileres\_por\_cliente}, \variable{duracion\_alquiler} y \variable{pelicula\_costo\_reposicion}:
  \[
    x'_i = \frac{x_i - \mu}{\sigma}
  \]

  \item \textbf{Normalización Min-Max} al intervalo $[0,1]$ para las cinco variables con rango conocido y sin valores extremos significativos:
  \[
    x'_i = \frac{x_i - x_{\min}}{x_{\max} - x_{\min}}
  \]

  \item \textbf{One-Hot Encoding (OHE)} para las columnas categóricas nominales:
  \begin{itemize}
    \item \variable{categoria\_nombre}: 16 géneros cinematográficos $\rightarrow$ 16 columnas binarias.
    \item \variable{pelicula\_clasificacion}: 5 niveles de clasificación de edad $\rightarrow$ 5 columnas binarias.
  \end{itemize}
\end{enumerate}

El vector de entrada final a la red neuronal tiene dimensión $\mathbf{x} \in \mathbb{R}^{31}$ (10 continuas escaladas + 16 géneros OHE + 5 clasificaciones OHE).

\chapter{Arquitectura de la Red Neuronal}
\label{cap:red}
% =============================================================================
\thispagestyle{fancy}

\section{Enfoque: Multi-Task Learning}

El aprendizaje multitarea (\ingles{Multi-Task Learning}, MTL) es un paradigma en el que un único modelo aprende a resolver varias tareas de forma simultánea, compartiendo representaciones internas. Esto ofrece dos ventajas principales:

\begin{enumerate}
  \item \textbf{Regularización implícita:} al optimizar múltiples funciones de pérdida a la vez, el gradiente de cada tarea actúa como regularizador para las demás, reduciendo el sobreajuste.
  \item \textbf{Eficiencia computacional:} una sola pasada hacia adelante (\ingles{forward pass}) produce ocho predicciones simultáneas en lugar de ocho pasadas independientes.
\end{enumerate}

\section{Arquitectura del modelo}

La arquitectura sigue el patrón \emph{cuerpo compartido + cabezas especializadas}:

\subsection{Capa de entrada}

Un vector $\mathbf{x} \in \mathbb{R}^{31}$ que concatena:
\begin{itemize}
  \item 10 variables continuas escaladas (Z-score y Min-Max).
  \item 16 indicadores de género OHE.
  \item 5 indicadores de clasificación de edad OHE.
\end{itemize}

\subsection{Cuerpo compartido}

$N$ bloques apilados secuencialmente, donde cada bloque consiste en:
\[
  h^{(\ell)} = \text{Dropout}_p\!\Bigl(\text{BN}\!\bigl(\text{ReLU}(W^{(\ell)} h^{(\ell-1)} + b^{(\ell)})\bigr)\Bigr)
\]

con $\text{BN}$ = Normalización por lotes (\ingles{Batch Normalization}), $p$ = tasa de \ingles{dropout} y $W^{(\ell)}, b^{(\ell)}$ los parámetros aprendidos de la capa $\ell$.

\subsection{Cabezas de salida}

Desde la última representación compartida $h^{(N)}$ se ramifican ocho cabezas independientes:

\begin{description}
  \item[4 salidas binarias] $\hat{y}_k = \sigma(w_k^\top h^{(N)} + b_k)$, donde $\sigma$ es la sigmoide. Función de pérdida: \ingles{binary cross-entropy}.
  \item[4 salidas multiclase (3 clases)] $\hat{\mathbf{y}}_k = \text{softmax}(W_k h^{(N)} + \mathbf{b}_k)$. Función de pérdida: \ingles{sparse categorical cross-entropy}.
\end{description}

La función de pérdida total es la suma de las ocho pérdidas individuales:
\[
  \mathcal{L}_{\text{total}} = \sum_{k=1}^{4} \mathcal{L}_{\text{BCE}}^{(k)} + \sum_{k=5}^{8} \mathcal{L}_{\text{SCE}}^{(k)}
\]

El optimizador utilizado es \textbf{Adam} con sus hiperparámetros por defecto ($\alpha = 10^{-3}$, $\beta_1 = 0.9$, $\beta_2 = 0.999$).

\section{Protocolo de entrenamiento}

\begin{itemize}
  \item \textbf{División de datos:} 80\% entrenamiento / 20\% testeo con semilla fija (\texttt{random\_state=42}) para garantizar reproducibilidad. Del 80\% de entrenamiento, el 10\% se reserva como conjunto de validación durante el ajuste.
  \item \textbf{Parada anticipada:} \ingles{EarlyStopping} con \texttt{patience=5} monitoreando \texttt{val\_loss} y restaurando los pesos del mejor epoch.
  \item \textbf{Métrica de selección:} precisión promedio (\ingles{accuracy}) sobre las ocho salidas en el conjunto de testeo.
  \item \textbf{Modelo principal:} la arquitectura con mayor \ingles{accuracy} promedio se guarda como \texttt{modelo\_principal.keras} para su uso en producción.
\end{itemize}

\section{Descripción general}

La aplicación web implementada en \texttt{paso\_5\_aplicacion\_web/app.py} provee una interfaz de predicción en tiempo real. El usuario ingresa los datos de una transacción de alquiler en un formulario HTML y el sistema devuelve instantáneamente las ocho predicciones del modelo.

\section{Flujo de predicción}

El proceso de inferencia sigue estos pasos dentro del servidor Flask:

\begin{enumerate}
  \item El formulario envía los datos crudos vía \texttt{POST} en formato JSON.
  \item Se reconstruye el vector de entrada:
  \begin{enumerate}
    \item Las cinco variables Z-score se transforman con \texttt{escalador\_zscore} cargado desde \texttt{.joblib}.
    \item Las cinco variables Min-Max se transforman con \texttt{escalador\_minmax}.
    \item Las columnas de género y clasificación de edad se construyen como vectores OHE alineados con el orden usado en el entrenamiento.
    \item Se concatena el vector final $\mathbf{x} \in \mathbb{R}^{31}$.
  \end{enumerate}
  \item Se llama a \texttt{modelo\_principal.predict(x.reshape(1,-1))} obteniendo ocho salidas.
  \item Las salidas se decodifican con los umbrales y mapas de clase correspondientes y se retornan como JSON al cliente.
\end{enumerate}

\section{Categorías de entrada soportadas}

\begin{itemize}
  \item \textbf{Géneros cinematográficos} (16): Acción, Animación, Ciencia ficción, Clásicos, Comedia, Deportes, Documental, Drama, Extranjero, Familia, Horror, Juegos, Música, Niños, Nuevo, Viajar.
  \item \textbf{Clasificaciones de edad} (5): G, NC-17, PG, PG-13, R.
\end{itemize}

La aplicación se inicia con \texttt{python paso\_5\_aplicacion\_web/app.py} y es accesible en \texttt{http://127.0.0.1:5000/}.

\chapter{Conclusiones y Trabajo Futuro}
\label{cap:conclusiones}
% =============================================================================

\section{Conclusiones}

\begin{enumerate}
  \item \textbf{Pipeline robusto y reproducible:} la arquitectura de cinco pasos con semillas fijas y escaladores persistidos garantiza que cualquier ejecución del sistema produce resultados idénticos, condición fundamental para la reproducibilidad científica.

  \item \textbf{Ingeniería de características con valor predictivo:} las variables derivadas (\variable{veces\_alquilada\_pelicula}, \variable{variedad\_generos\_cliente}, \variable{pelicula\_popularidad\_actores}) enriquecen significativamente el espacio de características respecto a los datos crudos de MongoDB.

  \item \textbf{El EDA reveló no-normalidad generalizada:} los test KS y los gráficos Q-Q confirman que la mayoría de las variables no siguen una distribución gaussiana, lo que valida el uso combinado de Z-score y Min-Max en lugar de una única estrategia de escalado.

  \item \textbf{Aprendizaje multitarea adecuado para el dominio:} predecir simultáneamente ocho indicadores de negocio correlacionados es conceptualmente correcto bajo el enfoque MTL, pues las tareas comparten información sobre el comportamiento del cliente y las características de la película.

  \item \textbf{Comparación de cinco arquitecturas:} la experimentación sistemática con variantes de profundidad, ancho y dropout permite seleccionar objetivamente la configuración óptima en lugar de depender de intuición.
\end{enumerate}

\section{Trabajo futuro}

\begin{itemize}
  \item Incorporar métricas de evaluación adicionales por tarea: F1-score, AUC-ROC para las salidas binarias y F1 macro para las multiclase.
  \item Implementar reentrenamiento incremental a medida que llegan nuevas transacciones.
  \item Explorar técnicas de \ingles{hyperparameter tuning} automático (Optuna, Keras Tuner) para encontrar arquitecturas más eficientes.
  \item Añadir explicabilidad con SHAP o LIME para que el personal de la videotienda pueda interpretar las predicciones individuales.
  \item Contenerizar la aplicación con Docker para simplificar el despliegue en entornos de producción.
\end{itemize}
